# Integration Guide: Phase 0 Context + Execution Workflow

**Purpose:** Shows how the three files work together for autonomous agent execution.

---

## The Three Files & Their Roles

### File 1: `PHASE_0_CONTEXT_GATHERING.md`

**Role:** Helps agent discover and document task context

- **When used:** Before Phase 1
- **Who uses it:** Agent (autonomously fills out)
- **Output:** `PHASE_0_CONTEXT_[TASK_ID].md` (generated by agent, reviewed by you)
- **Contains:** Questionnaire, template, decision tree, quality gates

### File 2: `EXECUTION-CHECKLIST.md`

**Role:** Quick reference checklist for each phase

- **When used:** During all 7 phases
- **Who uses it:** Agent (follows checklist step-by-step)
- **Output:** PR, tests, docs, journal
- **Contains:** Phase-by-phase tasks, quality gates, abort conditions

### File 3: `EXECUTION-IMPROVED.md`

**Role:** Full detailed workflow with examples

- **When used:** Reference during implementation
- **Who uses it:** Agent (consults when stuck on "how do I do X?")
- **Output:** Development journal (structured markdown)
- **Contains:** 7 phases, detailed examples, commands, troubleshooting

## Quick Start: Agent's Daily Workflow

### Morning: Agent Analyzes Repo & Finds Work

```bash
# 1. Check for work items
git log --oneline -20              # Recent commits, TODOs?
gh issue list --state open         # Open issues
gh pr list --review-requested       # Code review requests

# 2. Identify highest-priority item
# Example: "Dashboard crashes with 1000+ scores"
```

### Mid-Morning: Agent Fills Out Phase 0

```bash
# 1. Open PHASE_0_CONTEXT_GATHERING.md
# 2. Answer questionnaire (Part 1):
#    - Q1: What is the work item?
#    - Q2: 3-sentence summary?
#    - Q3: Acceptance criteria?
#    - Q4: Dependencies?
#    - Q5: Risks?

# 3. Fill out template (Part 2):
#    - Requirement, criteria, files, tests, risks

# 4. Check quality gates (Part 4):
#    - Clarity ✓, Testability ✓, Scope ✓, etc.

# 5. Generate output file:
#    File: .github/phase-0/PHASE_0_CONTEXT_DASHBOARD_CRASH.md
```

### Late Morning: Human Reviews (Optional)

```
✓ Human reads agent's Phase 0 context
✓ Asks clarifications or approves as-is
✓ Agent updates file based on feedback
```

### Afternoon: Agent Executes (7 Phases)

```bash
# Reference 1: PHASE_0_CONTEXT_DASHBOARD_CRASH.md
#   ↓ "What am I building?"

# Reference 2: EXECUTION-CHECKLIST.md
#   ↓ "Phase 1: Planning. What's my checklist?"
#   ├─ [ ] Files to modify identified
#   ├─ [ ] API changes documented
#   ├─ [ ] Test strategy outlined
#   └─ [ ] Rollback plan exists
#
#   ↓ "Phase 2: TDD Setup. What's my checklist?"
#   ├─ [ ] Test file created
#   ├─ [ ] Tests failing (red state)
#   └─ [ ] Coverage baseline checked
#
#   ... (repeat for all 7 phases)

# Reference 3: EXECUTION-IMPROVED.md (if stuck)
#   ↓ "How do I write a scaffold test? Show me an example."
#   ├─ Example scaffold test
#   └─ Explanation of TDD flow
```

### End of Day: Output Development Journal

```bash
# Agent generates:
# DEVELOPMENT_JOURNAL_DASHBOARD_CRASH.md
#
# Sections:
# 1. Context & Planning
# 2. Development Methodology
# 3. Implementation Deep Dive
# 4. Quality Assurance
# 5. CI/CD Pipeline
# 6. Knowledge Artifacts
# 7. Metrics & Retrospective

# File lives in: .github/journals/DEVELOPMENT_JOURNAL_DASHBOARD_CRASH.md
```

```
   Clarity: ✓ | Testability: ✓ | Scope: ? (unclear if DB migration needed)

3. PHASE_0_CONTEXT_DEBT_UPGRADE_JEST.md
   Clarity: ✓ | Testability: ✓ | Scope: ✓ | Dependencies: ✓ | Risks: ✓

Context #2 needs clarification. Can you confirm:
'Does fixing the dashboard crash require a database schema change?'

Once I hear back, I'll proceed with execution."

Human responds:
"Good work. For #2: No database change needed.
And for #1 & #3: Both are ready. Proceed."

Agent continues:
"Thanks. Updated PHASE_0_CONTEXT_BUG_DASHBOARD_CRASH.md.
All 3 are now READY FOR PHASE 1.
Proceeding with execution of #2 (highest priority bug)..."
```

### Checkpoint 2: During Phase 5 (CI/CD) If Tests Fail

```
Agent output:
"Phase 5 blocker: CI tests failing.

Error:
  FAIL: syncScore test - fetch is not defined

Root cause:
  Mock not set up for fetch in test setup

I can fix this by:
Option A: Add jest.mock('node-fetch') to test setup
Option B: Use a real fetch library (node-fetch or isomorphic-fetch)

Which approach? (A is faster, B is more robust)"

Human responds:
"Use Option A. Let's move fast."

Agent continues:
"Done. Committed fix, re-running CI...
All tests now passing ✓"
```

### Checkpoint 3: If Risk Changes Mid-Implementation

```
Agent output:
"Risk escalation: During Phase 3 implementation, I discovered:

Original risk: Race condition on 10x clicks
New risk: Database getting hammered with 100 syncs/second

Mitigation options:
1. Add debounce to 1 sync per second (current plan)
2. Add server-side rate limiting (more robust)

Recommendation: Add server-side rate limiting
(won't rely on client-side debounce alone)

Proceed? (adds ~1 hour to timeline)"

Human responds:
"Yes, add server-side rate limiting. Better safe."

Agent continues:
"Done. Adding rate limit middleware...
Updated Phase 0 context file with new risk mitigation."
```

## Tips: Getting the Most Out of These Files

### Tip 1: Store Phase 0 Files Permanently

```
Phase 0 context files are decision records.
Keep them in git (.github/phase-0/PHASE_0_CONTEXT_*.md)
Future developers can read them to understand "why" decisions were made.
```

### Tip 2: Use Phase 0 for Cross-Team Alignment

```
Before agent starts Phase 1:
- Share Phase 0 context with backend team / product manager
- They can spot blockers early
- No surprises during PR review
```

### Tip 3: Reference Phase 0 During Code Review

```
Reviewer: "Why debounce to 1/sec instead of immediate?"
You: "See .github/phase-0/PHASE_0_CONTEXT_SCORE_SYNC.md → Risks section"
Reviewer: "Ah, makes sense. Risk mitigation, not arbitrary choice."
```

### Tip 4: Track Metrics Over Time

```
Every development journal = one data point
After 10 features:
- Average time per phase
- Common blockers
- Pattern of what works
→ Continuously improve the workflow
```

### Tip 5: Use Phase 0 for Onboarding

```
New developer joins?
Show them a Phase 0 context file.
"This is how we think about requirements."
Much faster than explaining verbally.
```

## Troubleshooting: Common Issues

### Issue 1: Agent's Phase 0 Context is Vague

**Root cause:** Agent didn't follow decision tree or didn't ask for clarification

**Fix:**

```
Agent re-reads Part 3: Decision Tree
Agent re-checks Part 4: Quality Gates

If any gate fails:
- Clarity? Ask: "Can you describe in 3 sentences without 'improve'?"
- Testability? Ask: "Is each criterion measurable?"
- Scope? Ask: "What files will you modify?"

If still unclear, agent asks you for clarification.
```

### Issue 2: Execution Started but Phase 0 Incomplete

**Root cause:** Agent skipped Phase 0 quality gates

**Fix:**

```
STOP.
Return to Phase 0 context file.
Check all 6 quality gates:
- Clarity ✓?
- Testability ✓?
- Scope ✓?
- Dependencies ✓?
- Risks ✓?
- Readiness ✓?

If any fails, don't proceed. Fix Phase 0 first.
```

### Issue 3: Execution Blocked by High-Risk Item

**Root cause:** Phase 0 identified a HIGH risk with no mitigation

**Fix:**

```
In Phase 0 context, you should have escalated.
Example:
"Risk: HIGH (database migration affects 1M records)
Mitigation: NONE AVAILABLE
→ This work cannot proceed without mitigation plan"

If you missed this, go back to Phase 0, design mitigation,
then proceed with Phase 1.
```
